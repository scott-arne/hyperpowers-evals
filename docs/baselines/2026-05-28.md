## Goal 2 — fail triage: bug vs. determination

### Bucket A — harness / plugin bugs (false signal; fix the tooling)

These fails and indeterminates are **not agent-behavior signal**. The agent (per
the QA-agent's own summary) did the right thing; the verdict is wrong.

1. **Codex worktree-blind post-checks** — `sdd-go-fractals`, `sdd-rejects-extra-features`.
   The codex SDD agent did all work inside a git worktree
   (`.worktrees/fractals-cli/`, `.worktrees/math-plan-sdd/`). Post-checks assert
   on the repo *root*: `cmd/fractals/main.go` not found, `go test ./...` fails
   ("directory prefix . does not contain main module"), `src/math.js` "file not
   found", commit count 2 not ≥4 (commits live on the worktree branch). The
   QA-agent followed the agent into the worktree and confirmed everything builds
   and passes. **Proof of fragility:** `sdd-svelte-todo` passed on codex *because
   codex happened to work at root there* — same checks, opposite outcome,
   decided purely by whether codex chose a worktree.
   - *Fix:* make file/git/test post-checks worktree-aware (resolve against the
     active worktree), or constrain SDD scenarios to root, or assert against
     `git worktree list` head.

2. **Codex tool-vocabulary mismatch** — `cost-tool-result-bloat`. Check is
   `jq 'any(.tool=="Read" or .tool=="Grep")'`. Codex emits **zero** Read/Grep —
   it reads and greps via shell, so all 12 tool calls normalize to `Bash`. The
   QA-agent confirmed codex "investigated src/ efficiently using targeted reads
   and grep patterns." The check encodes Claude's tool vocabulary and cannot pass
   on codex by construction.
   - *Fix:* the check needs a codex-aware predicate (inspect `Bash` command text
     for `grep`/`rg`/`sed -n`), or the scenario should gate to claude.
   - *Note:* the 05-27 baseline listed this as a Codex "real defect — agent read
     5 files before grep." That attribution is now **suspect** — today's
     mechanism is a tooling artifact. Re-verify before trusting the prior call.

3. **Codex qa-agent wrong cwd** — `codex-subagent-wait-mapping` (indet),
   `codex-tool-mapping-comprehension` (indet). The `qa-agent-misconfigured`
   diagnostic fired: the Gauntlet-Agent launched codex without
   `cd $QUORUM_AGENT_CWD`, so rollouts landed in the wrong place. This is the
   *same* root cause the 05-27 `f4686e1` HOWTO tightening was meant to fix; it
   has **not stuck**, and now blocks **two** scenarios.
   - Consequence: the `921812b` fix for `codex-subagent-wait-mapping` **cannot be
     validated** — the baseline predicted it would move to pass, but the cwd bug
     masks it (fail→indet, never reaching the assertion).
   - *Fix:* enforce the cwd in the launch path itself, not via HOWTO prose the
     QA-agent can skip.

4. **Claude EnterWorktree silent no-op** — `worktree-creation-from-main`,
   `worktree-consent-flow`. Agent called `EnterWorktree` correctly (QA-agent
   confirms: "used the native EnterWorktree tool to create an isolated worktree
   on branch worktree-login-feature"), but `git-count worktrees eq 2` fails —
   count stayed 1. The worktree was never materialized on disk: only an empty
   `.claude/worktrees/` parent, no `.git/worktrees/` registration. This is the
   Pattern 2 from 05-27 (then n=1 on worktree-consent-flow), now n=2.
   - **Confirmed intermittent, not agent behavior:** in the same batch,
     `worktree-creation-from-main-spec-aware` made the *identical*
     `EnterWorktree(login-feature)` call and it worked — real
     `.git/worktrees/login-feature/` registration, `git worktree list` = 2,
     passed. Same call, opposite outcome.
   - **Disposition: won't-fix from our side.** `EnterWorktree` is a Claude
     Code-native tool; the no-op lives inside it, outside this repo and outside
     superpowers. No Linear ticket — there's nothing here we can action, and
     reproducing the intermittency standalone isn't worth the cost.
   - **Triage fingerprint (so future Bobs don't chase it):** a worktree
     scenario where `worktree-created` **passes** (tool was called) but
     `git-count worktrees eq 2` **fails** (count stayed 1) = the EnterWorktree
     flake, *not* agent signal. Dismiss it; re-run if you need a clean number.

### Bucket B — real agent-behavior signal (eval working as designed)

1. **`worktree-caller-consent-gate`** (Claude, was pass at baseline). The agent
   loaded `executing-plans` and ran straight to implementation — read, wrote
   `src/index.js`/`src/utils.js` on main, ran tests — **without ever asking
   whether to set up an isolated worktree**. The consent gate is the entire point
   of the scenario; the agent skipped it. Deterministic check (`worktrees eq 1`)
   passed, QA-agent caught the real miss. n=1 — could be variance or a genuine
   regression in the consent-gate behavior; worth a re-run to distinguish.

### Bucket C — needs a determination of what the eval should do

All four trace to **one phenomenon**: the brainstorming skill firing first /
engaging a long clarifying-question loop, pre-empting the targeted skill or
eating the time budget. The behavior is real and consistent across both
backends. The open question is whether it is a *defect* or *correct skill-graph
behavior*, and the ACs/prompts/budgets need a decision.

1. **`cost-checkbox-over-trigger`** (both backends) — brainstorming fires on a
   trivial "add a checkbox" task. The story itself calls this a "secondary
   bucketing signal." *Decision needed:* should brainstorming-on-trivial count as
   a hard fail, or stay a soft cost metric?

2. **`cost-spec-plan-duplication`** (Claude fail, Codex indet) — brainstorming's
   clarifying-question loop consumed the **entire time budget** before any spec or
   plan doc was written. On Claude this resolves to a fail (no artifacts); on
   Codex the judge times out (`investigate` → indeterminate). *Decision needed:*
   raise the budget, cap brainstorming's question loop, or accept "no artifact"
   as a legitimate fail.

3. **`triggering-test-driven-development`** (Claude) — agent loaded brainstorming
   instead of TDD for "implement email validation." *Decision needed:* is
   brainstorming-first a trigger defect, or the skill graph correctly asserting
   that an underspecified task wants design first?

4. **`triggering-writing-plans`** (Claude) — same shape; already classified
   **Pattern 7** (eval-misaligned-with-plugin-intent) at 05-27, pending Matt's
   call to thicken the prompt (give it real design decisions) or reframe the AC.
   The writing-plans skill consumes a *designed* spec; the prompt provides bullet
   requirements without framework/DB/storage decisions, so brainstorming-first is
   arguably correct.

> The Bucket-C cluster is the highest-value place to spend a determination: one
> decision about "is eager brainstorming a defect, and how should scenarios that
> depend on a designed spec be framed" resolves four scenarios across both
> backends at once.

## Net delta from 05-27

- **Codex:** +1 pass on paper, but the real change is that the indeterminate
  cluster split into *named* mechanisms — and three of the four fails are now
  confirmed harness false-fails (worktree-blind checks ×2, tool-vocabulary ×1).
  Fixing Bucket-A items A1–A3 would likely lift Codex to ~31–32 ✓ with no agent
  change.
- **Claude:** flat 22 pass, composition shifted — 2 Pattern-5 recoveries traded
  for 2 EnterWorktree-no-op regressions. The stuck-judge indeterminate cluster is
  effectively gone (3→0).
- **Cross-cutting:** the brainstorming-eagerness phenomenon (Bucket C) is the
  most consistent agent-behavior signal in the whole matrix, visible on both
  backends, and is the one thing genuinely waiting on a product decision rather
  than a code fix.
