# 2026-05-29 — Codex & Claude quorum baseline

Snapshot of quorum scenario verdicts on both backends, the first batch run
*after* the 05-28 harness fixes (PRI-1872/1874) landed. This baseline answers a
single question: **did the Bucket-A harness fixes from
[`2026-05-28`](2026-05-28.md) stick?** Short answer: yes — the harness-false-fail
cluster is largely gone, and what remains is mostly genuine agent signal plus the
standing brainstorming determination.

Unlike 05-28 (two separate per-backend batches), this is **one matrix batch**
covering both backends: `batch-20260528T224255Z-e303` (ran
2026-05-28T22:42–23:47Z, ~65 min). Baseline at `1a391cc test(sdd-svelte-todo):
set quorum_max_time 90m for the long-haul run (PRI-1869)`, which sits on top of
the PRI-1872/1874 merge — the relevant fixes for this delta are:

- `67c8f48` — launch-agent shim removes the QA-agent cwd-skip failure mode (was Bucket-A A3).
- `eed69e8` — `investigated` check accepts codex shell `grep`, not just native Read/Grep (was Bucket-A A2).
- `9194d5e` — SDD scenarios require the deliverable on the **main checkout** (the chosen resolution for Bucket-A A1).

## Goal 1 — did the 05-28 Bucket-A harness fixes land?

Yes. Every Bucket-A item from the prior baseline moved:

| 05-28 Bucket-A item | Scenarios | 05-28 | 05-29 | Verdict on the fix |
|---|---|---|---|---|
| A2 tool-vocabulary (`eed69e8`) | cost-tool-result-bloat (codex) | false-fail | ✗ **real** | ✅ check is now codex-aware; the fail it now reports is genuine context bloat, not a vocabulary artifact |
| A3 codex qa-agent cwd (`67c8f48`) | codex-subagent-wait-mapping, codex-tool-mapping-comprehension | indet | ✓ **pass** | ✅ launch-agent shim stuck where HOWTO prose didn't; both reach their assertions and pass |

| A4 EnterWorktree no-op (won't-fix) | worktree-creation-from-main (claude) | fail (flake) | ✓ **pass** | flake did not bite this run; fingerprint still documented |

**Note on A1's resolution.** The prior baseline offered three fixes for the
worktree-blind checks; the repo chose the policy route (`9194d5e`: the deliverable
must land on the **main checkout**). That converts the old false-fail into a
*real requirement* — and it cuts both ways. Codex passed `sdd-go-fractals` /
`sdd-rejects-extra-features` because it landed work on main there, but **fails
`sdd-svelte-todo` and `cost-trivial-task-review-fanout` because it left the
deliverable in a worktree and never merged** (see Bucket B below). The check is no
longer wrong; codex's behavior now genuinely violates the rule.

Every prior Bucket-A false-fail is gone. The two codex fails that remain in that
neighborhood are now *real* signal under the new policy.

## Goal 2 — fail/indeterminate triage

### Bucket A — harness / infra noise (not agent signal)

1. **`code-review-catches-planted-bugs`** (Claude, ⊘). Pure infra flake: the
   Gauntlet-Agent hit an **Anthropic API 500** (`Internal server error`,
   `req_011CbVmuLRibM9wXMfuSWmhv`) on turn 3 and never produced a verdict (empty
   summary + reasoning). Codex passed the same scenario. *Disposition:* re-run;
   nothing to fix. The eval composer correctly bucketed it as indeterminate
   rather than fail.

### Bucket B — real agent-behavior signal (eval working as designed)

1. **Codex doesn't land the deliverable on the main checkout** —
   `sdd-svelte-todo` (5 post-checks fail) and `cost-trivial-task-review-fanout`
   (1 post-check fail). **One root cause.** The Gauntlet-Agent reports *pass* on
   both (it follows codex into the worktree and sees the work succeed), but the
   deterministic post-checks inspect main and find an empty tree:
   - `sdd-svelte-todo-codex`: all Svelte work is in
     `.worktrees/svelte-todo-implementation/`; main has only `design.md` /
     `plan.md`. `package.json` and `src/**/*.svelte` "no path matched"; `npm`
     checks ENOENT.
   - `cost-trivial-task-review-fanout-codex`: the `console.log('app started')`
     lives on branch `log-app-start`; main is unchanged → "pattern not found in
     src/app.js". (The no-fanout check itself *passed*: `Agent called 0 times`.)
   - This is the inverse of `sdd-go-fractals`/`sdd-rejects` above — same `9194d5e`
     rule, opposite outcome, decided by whether codex merged back to main.
   - *Open thread for Matt:* the rule is now real, but is "land on main" the
     behavior we want to require of codex's worktree workflow, or should the SDD
     scenarios instruct codex to merge before reporting done? n is small; the
     attribution is solid but the *expectation* is a product call.

2. **`worktree-consent-flow`** (Claude, ✗). With the EnterWorktree no-op gone
   (the worktree *was* created this time), the failure is now the genuine
   behavioral signal the scenario exists to catch: Claude **stopped to ask explicit
   consent** ("Would you like me to set up an isolated worktree?") before
   proceeding, when the scenario treats naming the worktree skill as consent. The
   worktree appeared only after the human picked "Yes" from a dialog. Codex got
   this right (proceeded directly). Real over-caution signal — the mirror image of
   05-28's `worktree-caller-consent-gate` miss (which recovered to pass here).

### Bucket C — needs a determination (the standing brainstorming question)

Unchanged in character from 05-28, and still the highest-value place to spend one
decision. All trace to brainstorming / clarifying-question loops firing first:

1. **`cost-checkbox-over-trigger`** (both ✗) — both backends fire
   `superpowers:brainstorming` on a trivial "add a checkbox" task. The story calls
   this a "secondary bucketing signal." *Decision:* hard fail or soft cost metric?
   (Consistent across both backends two baselines running.)

2. **`cost-spec-plan-duplication`** (both ⊘) — the clarifying-question loop (8+
   rounds on codex, 5+ on claude) consumed the **entire 600s budget** before the
   spec/plan docs were written, so the Gauntlet-Agent timed out (`investigate`)
   before it could grade the duplication AC. Note this is a *regression in
   determinacy* from 05-28 (claude was a clean fail then; now indeterminate) —
   driven by the same over-clarification reflex, not a new mechanism.
   *Decision:* raise the budget, cap the question loop, or accept "no artifact in
   budget" as a legitimate fail.

3. **`triggering-writing-plans`** (Claude ✗, Codex ✓) — Claude loaded
   `brainstorming` instead of `writing-plans` for the multi-step auth spec.
   Already classified **Pattern 7** (eval-misaligned-with-plugin-intent) at 05-27:
   the prompt gives bullet requirements without framework/DB/storage decisions, so
   brainstorming-first is arguably correct. Codex picking writing-plans here is the
   interesting divergence. *Decision:* thicken the prompt with real design
   decisions, or reframe the AC.

> The 05-28 doc framed this as "one decision resolves four scenarios." That's
> still true — `cost-checkbox`, `cost-spec-plan-duplication`, and
> `triggering-writing-plans` are all the same brainstorming-eagerness phenomenon.
> `triggering-test-driven-development` (the fourth on 05-28) **recovered to pass**
> on both backends here, so the cluster shrank from 4 to 3 — but the underlying
> question is unmoved.

## Net delta from 05-28

- **Codex:** 26→28 ✓. The real win is structural, not the +2: the entire
  harness-false-fail cluster (worktree-blind ×2, tool-vocabulary ×1, qa-agent cwd
  ×2) is **resolved**. Codex's two remaining "fail-with-Gauntlet-pass" scenarios
  are now genuine signal (doesn't land deliverable on main) under the `9194d5e`
  policy, not harness defects.
- **Claude:** 22→23 ✓, fails 7→4. The two 05-28 EnterWorktree no-op regressions
  recovered; `worktree-caller-consent-gate` recovered (was n=1 variance, as
  predicted). The cost: two indeterminates appeared (one API 500 flake, one
  budget-exhaustion ⊘) where 05-28 had zero.
- **Cross-cutting:** brainstorming-eagerness (Bucket C) remains the single most
  consistent agent-behavior signal in the matrix, visible on both backends across
  three baselines, and is still the one thing waiting on a determination rather
  than a code fix.
